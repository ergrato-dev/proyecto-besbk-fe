# Internacionalización (i18n) — NN Auth System

## Terminología fundamental

Antes de ver el código, es esencial entender los tres términos clave:

### i18n — Internacionalización

> **"i18n"** es una abreviatura: la palabra _internationalization_ tiene 20 letras,
> la `i`, luego **18 letras** en el medio, luego la `n` → `i18n`.

**¿Qué significa?** Preparar el código para que admita múltiples idiomas **sin
modificar el código fuente**. El texto de la interfaz no está hardcodeado en los
componentes — está en archivos separados llamados catálogos de traducción.

```typescript
// ❌ Texto hardcodeado — NO internacionalizable
function LoginPage() {
  return <h1>Iniciar sesión</h1>;  // siempre español, no se puede cambiar
}

// ✅ Internacionalizado — el texto viene de un catálogo
function LoginPage() {
  const { t } = useTranslation();
  return <h1>{t("auth.login.title")}</h1>;  // lee del JSON según el idioma activo
}
```

### l10n — Localización

> **"l10n"** = _localization_: la `l`, **10 letras** en el medio, la `n`.

**¿Qué significa?** Los archivos JSON con las traducciones específicas para cada
idioma. Si i18n es la "tubería", l10n es el "agua" que corre por ella.

```
fe/src/locales/
├── es/
│   └── translation.json   ← l10n español: "auth.login.title" = "Iniciar sesión"
└── en/
    └── translation.json   ← l10n inglés:  "auth.login.title" = "Sign in"
```

### locale

Un identificador estándar ISO 639-1 que combina **idioma** y opcionalmente **región**:

| Código  | Significado               | Ejemplos de uso             |
| ------- | ------------------------- | --------------------------- |
| `es`    | Español (sin región)      | Default del sistema         |
| `en`    | Inglés (sin región)       | Segundo idioma soportado    |
| `es-CO` | Español de Colombia       | Podría usarse en el futuro  |
| `en-US` | Inglés de Estados Unidos  | Podría usarse en el futuro  |

> El sistema usa locales **sin región** (`es`, `en`) por simplicidad pedagógica.
> Agregar variantes regionales es posible sin cambiar la arquitectura.

## El problema que resuelve

Sin i18n, cada texto de la interfaz está hardcodeado en el componente:

```typescript
// ❌ SIN i18n — para soportar inglés habría que duplicar TODOS los componentes
function RegisterPage() {
  return (
    <form>
      <h1>Crear cuenta</h1>
      <label>Correo electrónico</label>
      <input type="email" placeholder="tu@correo.com" />
      <label>Contraseña</label>
      <button>Registrarse</button>
    </form>
  );
}
```

Con i18n, **un solo componente** sirve para todos los idiomas:

```typescript
// ✅ CON i18n — mismo componente, cualquier idioma
function RegisterPage() {
  const { t } = useTranslation();
  return (
    <form>
      <h1>{t("auth.register.title")}</h1>
      <label>{t("auth.register.emailLabel")}</label>
      <input type="email" placeholder={t("auth.register.emailPlaceholder")} />
      <label>{t("auth.register.passwordLabel")}</label>
      <button>{t("auth.register.submit")}</button>
    </form>
  );
}
```

## Herramientas del sistema

| Librería                         | Versión | Propósito                                                   |
| -------------------------------- | ------- | ----------------------------------------------------------- |
| `i18next`                        | 26.0.3  | Motor principal de traducción                               |
| `react-i18next`                  | 17.0.2  | Integración con React: habilita `useTranslation()` y re-render automático al cambiar idioma |
| `i18next-browser-languagedetector` | 8.2.1  | Plugin que detecta el idioma del navegador automáticamente  |

### Relación entre las tres librerías

```
i18next (motor)
    │
    ├── i18next-browser-languagedetector  → detecta idioma antes de renderizar
    │                                       (localStorage → navigator → htmlTag)
    │
    └── react-i18next                     → conecta i18next con React
                                            · hook useTranslation()
                                            · React re-renderiza al cambiar idioma
```

| Código | Idioma              | Archivo de catálogo                    | Notas             |
| ------ | ------------------- | -------------------------------------- | ----------------- |
| `es`   | Español (por defecto) | `fe/src/locales/es/translation.json` | Fallback del sistema |
| `en`   | English             | `fe/src/locales/en/translation.json`   |                   |

> **Agregar un tercer idioma** (ej: portugués `pt`): crear
> `fe/src/locales/pt/translation.json`, importarlo en `fe/src/i18n.ts` y agregarlo
> al objeto `resources`. Sin cambios en los componentes.

## Configuración central: `fe/src/i18n.ts`

Este archivo es el punto de entrada de todo el sistema i18n. Se importa en
`main.tsx` antes de que cualquier componente se renderice.

```typescript
/**
 * Archivo: i18n.ts
 * ¿Qué? Configuración central de i18next para el sistema de idiomas.
 * ¿Para qué? Inicializar el motor de traducción con los recursos de cada idioma,
 *            el detector de idioma del navegador y el comportamiento de fallback.
 * ¿Impacto? Sin este archivo, react-i18next no sabría qué traducciones usar
 *           ni en qué idioma mostrarlas.
 */

import i18n from "i18next";
import LanguageDetector from "i18next-browser-languagedetector";
import { initReactI18next } from "react-i18next";

// ¿Qué? Importar los archivos JSON de traducción directamente (recursos estáticos).
// ¿Para qué? Cargar todas las traducciones en memoria al iniciar la app.
// ¿Impacto? Ambos idiomas disponibles al instante — sin peticiones HTTP adicionales.
import esTranslation from "./locales/es/translation.json";
import enTranslation from "./locales/en/translation.json";

const resources = {
  es: { translation: esTranslation },
  en: { translation: enTranslation },
};

i18n
  // 1. Detecta el idioma del usuario automáticamente
  .use(LanguageDetector)

  // 2. Conecta i18next con React (habilita useTranslation() y re-renders)
  .use(initReactI18next)

  // 3. Inicialización con la configuración completa
  .init({
    resources,

    // Idioma de respaldo cuando la clave no existe en el idioma activo
    fallbackLng: "es",

    // Namespace por defecto → permite t("clave") sin t("translation:clave")
    defaultNS: "translation",

    debug: false,  // en desarrollo: true para ver logs de claves faltantes

    // Orden de detección del idioma
    detection: {
      order: ["localStorage", "navigator", "htmlTag"],
      lookupLocalStorage: "i18nextLng",  // clave en localStorage
      caches: ["localStorage"],           // guardar el idioma detectado
    },

    interpolation: {
      escapeValue: false,  // React maneja el escape de HTML — evita doble escape
    },
  });

export default i18n;
```

## Flujo de detección de idioma al iniciar la app

```
Usuario abre la app por primera vez
          ↓
1. ¿Hay "i18nextLng" en localStorage?
   └── SÍ → usar ese valor (preferencia guardada por el usuario)
   └── NO → continuar ↓

2. ¿Tiene navigator.language configurado?
   └── SÍ → usar el idioma del sistema operativo/navegador
             (ej: "es-CO" → normaliza a "es")
   └── NO → continuar ↓

3. ¿Hay atributo lang en <html>?
   └── SÍ → usarlo
   └── NO → continuar ↓

4. FALLBACK → idioma: "es"
```

**Ejemplo práctico:**
- Usuario de Colombia con Chrome en español → `navigator.language = "es-CO"` → la app aparece en **español**
- Usuario de EE.UU. con Firefox en inglés → `navigator.language = "en-US"` → la app aparece en **inglés**
- Usuario que cambió manualmente a inglés antes → `localStorage["i18nextLng"] = "en"` → la app aparece en **inglés** aunque el navegador sea español

## Estructura del catálogo de traducción

Los catálogos son archivos JSON con **claves en inglés** (camelCase, agrupadas por
sección) y **valores en el idioma correspondiente**.

```json
// fe/src/locales/es/translation.json (fragmento)
{
  "auth": {
    "login": {
      "title": "Iniciar sesión",
      "emailLabel": "Correo electrónico",
      "passwordLabel": "Contraseña",
      "submit": "Iniciar sesión",
      "noAccount": "¿No tienes cuenta?",
      "registerLink": "Regístrate"
    },
    "register": {
      "title": "Crear cuenta",
      "submit": "Registrarse"
    }
  },
  "dashboard": {
    "welcome": "Bienvenido, {{name}}"
  },
  "nav": {
    "logout": "Cerrar sesión"
  },
  "common": {
    "loading": "Cargando...",
    "cancel": "Cancelar"
  },
  "language": {
    "label": "Idioma",
    "es": "Español",
    "en": "English"
  }
}
```

```json
// fe/src/locales/en/translation.json (fragmento)
{
  "auth": {
    "login": {
      "title": "Sign in",
      "emailLabel": "Email address",
      "passwordLabel": "Password",
      "submit": "Sign in",
      "noAccount": "Don't have an account?",
      "registerLink": "Sign up"
    }
  },
  "dashboard": {
    "welcome": "Welcome, {{name}}"
  },
  "nav": {
    "logout": "Sign out"
  }
}
```

### Convenciones de claves

| Prefijo               | Sección                               |
| --------------------- | ------------------------------------- |
| `auth.login.*`        | Textos de la página de login          |
| `auth.register.*`     | Textos del formulario de registro     |
| `auth.changePassword.*` | Textos del cambio de contraseña    |
| `auth.forgotPassword.*` | Textos de recuperación de contraseña |
| `auth.resetPassword.*`  | Textos del restablecimiento         |
| `dashboard.*`         | Textos del dashboard                  |
| `nav.*`               | Textos de la navbar                   |
| `common.*`            | Textos reutilizables (loading, etc.)  |
| `language.*`          | Textos del selector de idioma         |

## El hook `useTranslation()`

```typescript
import { useTranslation } from "react-i18next";

function LoginPage() {
  // ¿Qué? Hook que expone la función de traducción t() y el idioma activo.
  // ¿Para qué? Acceder a las traducciones dentro de cualquier componente React.
  // ¿Impacto? Cuando el idioma cambia, React re-renderiza TODOS los componentes
  //           que usen useTranslation() — la interfaz completa se traduce al instante.
  const { t, i18n } = useTranslation();

  return (
    <div>
      <h1>{t("auth.login.title")}</h1>

      {/* Interpolación de variables */}
      <p>{t("dashboard.welcome", { name: "Carlos" })}</p>
      {/* Renderiza: "Bienvenido, Carlos" (es) | "Welcome, Carlos" (en) */}

      {/* Idioma activo */}
      <p>Idioma actual: {i18n.language}</p>
    </div>
  );
}
```

## Cambio de idioma: el componente `LanguageSwitcher`

```typescript
// components/ui/LanguageSwitcher.tsx
import { useTranslation } from "react-i18next";

export function LanguageSwitcher() {
  const { t, i18n } = useTranslation();

  // ¿Qué? Cambia el idioma activo de la app.
  // ¿Para qué? i18next guarda "locale" en localStorage automáticamente,
  //            y luego sincroniza con el backend si el usuario está autenticado.
  // ¿Impacto? Todos los componentes con useTranslation() se re-renderizan.
  const handleChange = (locale: string) => {
    i18n.changeLanguage(locale);
    // Aquí también se llama a PATCH /api/v1/users/me/locale si hay sesión activa
  };

  return (
    <div>
      <span>{t("language.label")}: </span>
      <button
        onClick={() => handleChange("es")}
        className={i18n.language === "es" ? "font-bold" : ""}
      >
        {t("language.es")}
      </button>
      {" / "}
      <button
        onClick={() => handleChange("en")}
        className={i18n.language === "en" ? "font-bold" : ""}
      >
        {t("language.en")}
      </button>
    </div>
  );
}
```

## Persistencia de la preferencia de idioma

El sistema persiste la preferencia del usuario en **dos lugares**:

### 1. `localStorage` (frontend, sin sesión)

Cuando el usuario cambia de idioma, `i18next-browser-languagedetector` guarda
automáticamente el valor en `localStorage["i18nextLng"]`:

```
localStorage["i18nextLng"] = "en"
```

Esto funciona **sin estar autenticado**. Al recargar la página, el detector lee
esta clave primero y restaura el idioma elegido.

### 2. Columna `locale` en la base de datos (backend, con sesión)

Cuando el usuario está autenticado y cambia de idioma, el frontend también llama
al endpoint del backend para persistir la preferencia en la BD:

```
PATCH /api/v1/users/me/locale
Authorization: Bearer <access_token>
Content-Type: application/json

{ "locale": "en" }

→ 200 OK: { "locale": "en" }
```

Esto permite que la preferencia de idioma se recupere al iniciar sesión desde
un dispositivo diferente — el `locale` guardado en la BD precede al del `localStorage`.

### Esquema de la tabla `users` (Flyway V4)

```sql
-- V4__add_locale_to_users.sql
ALTER TABLE users
  ADD COLUMN locale VARCHAR(10) NOT NULL DEFAULT 'es';
```

```kotlin
// model/User.kt — campo locale
@Column(name = "locale", nullable = false, length = 10)
var locale: String = "es"

// dto/AuthDtos.kt — locale en la respuesta
data class UserResponse(
    val id: UUID,
    val email: String,
    val fullName: String,
    val isActive: Boolean,
    val isEmailVerified: Boolean,
    val locale: String,          // ← incluido para que el frontend pueda sincronizar
    val createdAt: Instant
)
```

## Flujo completo: usuario cambia de idioma

```
Usuario hace clic en "English" en el LanguageSwitcher
          ↓
1. i18n.changeLanguage("en")
   → i18next carga las traducciones del catálogo en/translation.json
   → React re-renderiza todos los componentes con useTranslation()
   → La interfaz cambia al inglés INMEDIATAMENTE (sin recarga)
          ↓
2. i18next-browser-languagedetector guarda en localStorage
   localStorage["i18nextLng"] = "en"
   → En la próxima visita, la app abre directamente en inglés
          ↓
3. Si el usuario está autenticado:
   PATCH /api/v1/users/me/locale  { "locale": "en" }
   → Se guarda en la BD: users.locale = "en"
   → Al iniciar sesión desde otro dispositivo, se restaura la preferencia
```

## Importación en `main.tsx`

```typescript
// main.tsx
// ¿Qué? Importar la configuración de i18n ANTES de renderizar cualquier componente.
// ¿Para qué? Garantizar que las traducciones estén disponibles desde el primer render.
// ¿Impacto? Si se importa después de App, algunos componentes podrían renderizarse
//           sin traducciones y mostrar las claves crudas (ej: "auth.login.title").
import "./i18n";   // ← debe ser una de las primeras importaciones

import { StrictMode } from "react";
import { createRoot } from "react-dom/client";
import App from "./App";

createRoot(document.getElementById("root")!).render(
  <StrictMode>
    <App />
  </StrictMode>,
);
```

## Tests: mock de i18n con Vitest

En los tests, `useTranslation()` debe ser mockeado para evitar dependencias del
sistema de idiomas:

```typescript
// test/setup.ts — configuración global de Vitest
import { vi } from "vitest";

// ¿Qué? Mock de react-i18next para que t("clave") devuelva la clave misma.
// ¿Para qué? Los tests validan estructura y comportamiento, no traducciones.
// ¿Impacto? Sin este mock, los tests fallan porque i18next no inicializa
//           correctamente en el entorno de Node.js (sin navegador).
vi.mock("react-i18next", () => ({
  useTranslation: () => ({
    t: (key: string) => key,    // t("auth.login.title") → "auth.login.title"
    i18n: {
      changeLanguage: vi.fn(),
      language: "es",
    },
  }),
  Trans: ({ i18nKey }: { i18nKey: string }) => i18nKey,
}));
```
